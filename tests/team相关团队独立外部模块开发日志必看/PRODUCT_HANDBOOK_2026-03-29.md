# UCLAW 产品手册（2026-03-29）

> 用途：给实施、运维、管理员和一线使用者提供一份可执行的使用说明。  
> 范围：基于当前一期主线能力编写。

---

## 1. 产品组成

当前主线由以下部分组成：

- Web 对话工作台
- 角色化协作会话
- Feishu 接入
- Scheduler 定时任务
- Skills / MCP 管理
- 记忆链与历史记录

---

## 4. 基础配置

### 4.1 最小环境变量

```dotenv
NODE_ENV=production
PORT=3001
CORS_ORIGIN=https://your-domain.example.com
UCLAW_DATA_PATH=.uclaw
UCLAW_API_BASE_URL=
UCLAW_API_KEY=
UCLAW_DEFAULT_MODEL=
```

### 4.2 说明

- `PORT`：服务端口
- `CORS_ORIGIN`：允许访问的前端域名
- `UCLAW_DATA_PATH`：运行数据目录，建议保持项目内相对路径
- `UCLAW_API_BASE_URL`：模型服务地址
- `UCLAW_API_KEY`：模型服务密钥
- `UCLAW_DEFAULT_MODEL`：默认模型

### 4.3 数据目录

默认运行数据目录：

- `./.uclaw/web`

建议保持在项目根目录内，便于统一管理和持久化挂载。

## 5. 页面与功能说明

### 5.1 首页 / 对话工作台

主要功能：

- 新建会话
- 选择角色进行工作
- 输入任务并开始执行
- 追加续聊
- 查看最近会话

适用场景：

- 日常提问
- 任务分配
- 快速生成内容
- 文件与资料辅助处理

### 5.2 会话历史

主要功能：

- 查看最近会话
- 重新进入会话
- 管理标题与状态

### 5.3 定时任务

主要功能：

- 创建定时任务
- 选择角色
- 设定任务提示词
- 启停任务
- 查看执行历史

适合：

- 日报整理
- 定时巡检
- 周期性摘要
- 自动提醒类任务

### 5.4 Skills

主要功能：

- 查看已安装技能
- 按需启用或配置技能
- 让角色获得特定能力

### 5.5 MCP

主要功能：

- 查看当前接入的工具服务
- 配置连接信息
- 管理可用能力入口

### 5.6 Settings

当前设置中心不仅是基础配置页，同时承担控制台角色。

主要配置区域包括：

- 模型与运行配置
- IM / 渠道配置
- 会话缓存与记忆
- 资源与备份
- 数据能力

## 6. 角色使用说明

### 6.1 当前固定角色

- `organizer`
- `writer`
- `designer`
- `analyst`

### 6.2 使用原则

- 角色是主入口
- 模型只是角色背后的运行配置
- 同一角色可在 Web 与 Feishu 之间共享连续性

### 6.3 典型分工

- `organizer`：任务整理、路线拆分、流程收敛
- `writer`：文字撰写、润色、摘要、说明稿
- `designer`：视觉表达、设计相关辅助
- `analyst`：信息分析、结构化归纳、研究整理

---

## 7. Feishu 使用说明

### 7.1 当前定位

Feishu 是当前现役渠道主线之一。

### 7.2 使用方式

- 在设置中完成 Feishu 配置
- 将渠道绑定到指定角色
- 由该角色承接消息与上下文

### 7.3 使用特点

- Feishu 与 Web 可共享同一身份记忆链
- 角色绑定后，渠道侧消息不再是孤立会话

## 8. Skills 与 MCP 配置说明

### 8.1 Skills

Skills 用于增加特定工作能力。

常见用途：

- 写作辅助
- 文件处理
- 外部平台协作
- 专项工作流

### 8.2 MCP

MCP 用于接入外部工具与服务。

常见用途：

- 工具调用
- 外部系统连接
- 结构化服务接入

### 8.3 配置建议

- 先确认角色定位，再决定启用哪些 Skills / MCP
- 不建议一次性把所有扩展能力都打开
- 应优先保证主链稳定，再逐步扩展

## 9. 文档与附件能力

当前系统底层支持常见文档解析，包括：

- `pdf`
- `doc`
- `docx`
- `txt`
- `md`
- `csv`
- `json`
- `xml`
- `html`
- `xlsx（基础）`

建议使用原则：

- 小文件优先直接处理
- 大文件在演示环境中建议拆分或分段上传
- 正式环境可结合后续独立文件处理链路扩展

## 10. 部署建议

### 10.1 推荐方式

优先级从高到低：

1. 传统 Linux 主机
2. Render Web Service
3. Zeabur Node 服务模式

### 10.2 不建议方式

- 直接将当前仓库作为完整运行时部署到 Vercel

原因：

- 当前运行模型依赖 Express
- 当前主链依赖 WebSocket
- 不是纯静态前端项目

## 12. 常见问题

### 12.1 启动后页面打不开

请检查：

- 服务是否真正启动
- 端口是否被占用
- 前端与后端地址是否一致

### 12.2 会话无法正常运行

请检查：

- API 基础地址
- API Key
- 默认模型是否可用

### 12.3 渠道消息没有进入系统

请检查：

- 渠道配置是否完成
- 角色是否正确绑定
- 后端对应渠道运行链是否已启动

### 12.4 附件处理异常

请优先确认：

- 文件类型是否属于当前支持范围
- 文件体积是否过大
- 演示环境是否启用了保守限制

## 13. 面向客户的使用建议

如果面向客户进行演示或试用，建议按以下顺序介绍：

1. 先展示多角色协作入口
2. 再展示历史连续性
3. 再展示定时任务
4. 最后展示 Skills / MCP 等扩展能力

这样更容易让客户先建立对主线价值的理解，再理解扩展空间。
