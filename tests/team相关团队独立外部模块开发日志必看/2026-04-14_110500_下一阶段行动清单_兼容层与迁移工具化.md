## 一、迁移流程工具化：当前问题

### 当前痛点

迁移流程虽然已经跑通，
但现在还严重依赖人工步骤：

- 手工抽检查包
- 手工整理完整源码包
- 手工复制到新地盘
- 手工 `npm ci`
- 手工 `npm run dev:web`
- 手工看日志判断缺件
- 手工执行路径重写脚本
- 手工跑导出状态接口
- 手工写 1 / 2 / 3 / 4 验收记录

这意味着：

- 方法是对的
- 但还没变成低成本、稳定可重复的流程

## 最值得先工具化的步骤

### Step A

把“检查包 + 完整包”整理成一条统一脚本：

- 产出 runtime focus bundle
- 产出 full source package
- 自动写 manifest

当前状态：

- `scripts/migration-rehearsal.mjs` 已经落下这一步的主干能力

### Step B

把“复制到新地盘 + 安装依赖 + 起服务 + 健康检查”做成一条脚本：

- 创建 runcheck sandbox
- `npm ci`
- `npm run dev:web`
- 收 stdout / stderr / runtime file / health

当前状态：

- `scripts/migration-rehearsal.mjs` 已经能完成这一步

### Step C

把“配置路径重写 + 最小种子补齐 + export 状态查询”做成一条脚本：

- 种最小 `app_config`
- 重写路径型配置
- 跑 `/api/role-runtime/:roleKey/exports`
- 回收结果

当前状态：

- `scripts/migration-rehearsal.mjs` 已经落下
  - 最小 `app_config` 种子
  - 路径重写
  - runcheck 启动与 `/health` 验收
- 下一步主要是把 export 状态查询也并进同一条链

### Step D

把“1 / 2 / 3 / 4 验收记录”变成半自动模板：

- 不是让模型每次重新写一篇大文章
- 而是按结果填空生成

## 二、兼容层减法：当前问题

### 现在不能硬删的部分

#### 1. `legacyAttachment*` fallback

位置：

- `server/libs/httpSessionExecutor.ts`
- `server/remote_httpSessionExecutor.ts`

原因：

- 旧的 `<role>/attachments/manual`
- 旧的 `<role>/attachments/exports`

里的文件还要靠它们兜底读取。

如果直接删，
会立刻导致旧文件读不到。

#### 2. `exportVerification.roots.legacy`

位置：

- `server/libs/exportVerification.ts`

原因：

- 旧 exports 目录里的结果还需要被状态接口看见
- 没迁干净之前不能直接撤掉

#### 3. `attachmentRuntime` 的 projectRoot 锚定

位置：

- `server/libs/attachmentRuntime.ts`

原因：

- 这是当前“输入文件路径只能在 projectRoot 内”的真相门口
- 不能误删

## 现在可以继续减的部分

### 1. `getRoleHomeContext` 的 legacy 字段

虽然内部 fallback 还要保留，
但 context 对象里准备的 `legacyAttachment*` 字段，
已经不应该继续作为 agent-facing 概念存在。

### 2. `runRoleHomeFilesTool` 的 `legacyEntries`

现在代码里还会算，
但已经不再回给 agent。

这是很适合继续减掉的一层“内部噪音”。

### 3. `exportVerification` 的 legacy 枚举方式

现在它默认合并 primary + legacy。

后面可以继续减到：

- 只有 legacy 根下真的有文件时，才显式带 legacy entries

### 4. 旧兼容注释

很多地方虽然逻辑没错，
但还保留旧时代的注释或说明语气。

这些会继续增加理解成本，
应该持续清扫。

## 三、建议顺序

### 第一优先

先做迁移流程工具化的 `A + B`

原因：

- 这直接降低重复劳动
- 也能减少“每次又是人肉搬运”的疲劳

补充：

- 现在 `A + B` 已经不是待开始项，而是已有脚本底座
- 下一步应该从“补齐结果采样和验收模板”继续往前推

### 第二优先

再做兼容层减法里的：

- `getRoleHomeContext` legacy 字段瘦身
- `runRoleHomeFilesTool` legacyEntries 清理

原因：

- 这些已经不影响旧文件读取
- 但能明显降低理解负担

### 第三优先

最后再动：

- `exportVerification` legacy 枚举策略
- 更深层的 fallback 退场

因为这一步更接近真正“退兼容”，
需要确认旧数据迁移情况。

而是方向已经很清楚：

- **迁移流程要工具化**
- **兼容层要分层退场**

这两件事做好，
