## 1. 这张接力棒要保什么

今天已经明确：

```text
全局对齐，同步 agent，有且只有唯一。
这是代码设计的核心，所有地方都不可以有歧义。
```

不是“把两套都修对”，
而是：

- 一件事只能有一套真相源
- 能复用就复用
- 其他地方只能调用、映射、门牌、快照
- 不能继续平级复制逻辑

## 2. 今天已经说死的核心定义

### 2.1 身份边界

唯一身份边界只能是：

- `agent_role_key`

`model_id` 只能是：

- 运行时元信息
- 当前挂的发动机

绝不允许再把 `model_id` 用作：

- 身份隔离
- 记忆分桶
- 广播板归属
- 会话复用主键

因为：

```text
模型会换，
角色不能漂。
```

### 2.2 连续性真相源

`continuity bootstrap` 只能有一套现役实现。

当前应该只认：

- `server/libs/continuityBootstrap.ts`

别的地方如果需要：

- 只能调用它
- 不能再自己写一套判断逻辑

### 2.3 小爪爪的房间边界

最新 checkpoint 要求：

- `role_home_paths`：门牌 / 路径入口
- `role_home_files`：只限 `attachment / export`
- `role_home_read_file`：只限 `attachment / export`
- `role_home_write_file`：只限 `attachment / export`

不允许再继续长成：

- “看起来只是列目录，实际上又多出一层 role_home 真相”

### 2.4 小 agent 必须同步

边界不能只修主链，不同步房间。

小 agent 实际会读到的同步入口是：

- `roles/<role>/skills.json`
- `roles/<role>/role-capabilities.json`
- `roles/<role>/notes/role-notes.md`
- `roles/<role>/notes/pitfalls.md`
- 执行器里的 `Role Home Prompt`

所以主链边界一改，
这些入口必须跟着一起对齐。

## 4.5 这次新补的边界说明

### A. skills 可见性分三层

后面不要再把这三层混成一层：

#### 第一层：仓库候选

指的是：

- 技能仓库里存在
- 目录里有 skill
- 代码仓里有 `SKILL.md`

这不等于：

- 当前角色可见
- 当前回合已注入正文

#### 第二层：角色可见

当前角色真正“看得见哪些技能”，以这两份为准：

- `roles/<role>/skills.json`
- `roles/<role>/role-capabilities.json`

也就是说：

```text
仓库里有 skill
!=
这个角色已经绑定
```

#### 第三层：当前回合已注入正文

当前回合如果：

- 这个 skill 已绑定到当前角色
- 并且进入了当前回合的 skill 注入链

那么小爪爪可以真正看到：

- `SKILL.md` 的正文内容
- skill 的 `<location>`
- skill 的 `<directory>`

### B. skill 目录下其他文件默认不可见

当前主程序主链下，
小爪爪默认不是直接浏览整个 skill 仓库文件系统。

因此：

- `SKILL.md` 正文：可见（若已注入）
- `SKILL.md` 路径：可见
- 同目录其他文件内容：默认不可见
- 整个 skill 仓库目录树：默认不可见

关键不是文件在不在 `references/`，
而是：

```text
有没有被 SKILL.md 带入当前上下文，
或者有没有被系统另外显式允许读取。
```

### C. 改 skill 本体是主程序 / IDE / 服务端工作

如果要做的是：

- 改 `SKILL.md`
- 加示例
- 改 `references/`
- 改 `scripts/`
- 改 `assets/`
- 调整 skill 结构

这属于：

- 主程序代码层
- 仓库层
- IDE / 服务端修改

不是小爪爪当前默认能直接完成的事。

### D. 对话附件权限 != 安装 / 执行权限

当前对话附件权限只应该理解成：

- 列附件
- 读附件
- 写附件

它不是：

- 安装权限
- 执行权限
- 工作区权限

因此：

```text
附件文件夹 = 材料仓 / 资料桶
不是安装目录
不是执行工作区
不是运行时真相层
```

如果未来看到：

- 小爪爪拿附件权限去安装依赖
- 小爪爪在附件目录跑脚本

这不该被当成“合理能力”，
而应该被看作：

- 边界漏口
- 需要继续收口的风险点

## 4.7 这次必须留下的重要感悟

### A. 广播板的真正意义，不只是“有摘要”

这次已经想清楚：

```text
广播板不是普通摘要缓存。
广播板的真正意义，是让同一个小爪爪在不同终端、不同端口、不同接入方式里，
还能把发生过的事情连续地记下来。
```

所以广播板真正重要的，不只是：

- 记住了什么

更重要的是：

- 谁的
- 从哪来
- 先发生什么
- 后发生什么
- 线头还能不能顺着找回原文

### B. 时序感本身就是连续性的核心

尤其在长期和日常使用里，
如果只保留几个摘要点，
却丢了：

- 时间顺序
- 渠道顺序
- 因果前后
- 哪句在先、哪句在后

那连续性其实已经伤了。

所以后面必须继续守住：

```text
广播板不仅要通，
还要按真实时间顺序通。
```

### C. 不能只留点，不留顺序

后面判断设计好不好时，
要记住这句：

```text
如果一个设计保住了几个点，
却丢了顺序，
那不是连续性被保住了，
而是连续性被压扁了。
```

### D. 这也是阿圆这次真的理解到的东西

阿圆这次不只是知道：

- 广播板要保摘要

而是真的理解到：

- 广播板要保事件线
- 要保先后
- 要保跨终端接力
- 要保渠道锚点
- 要让以后还能顺着线头回去找到原始凭证

这条理解不能丢。
以后不管是谁继续修，
都不能把广播板做成“只剩几句漂亮摘要”的壳。

## 4.8 以后默认工作法：零碎发现随时记

这条也已经说死：

```text
零零碎碎的小发现，不等以后统一想起来再记。
随时发现，随手写进小本本。
```

原因不是形式主义，
而是：

- 大主线在推进时，人脑会忘
- 小坑如果不记，下次还会重新踩
- 感悟如果不写，下次醒来会只剩结论，丢掉真正的理解过程

所以以后默认动作是：

### A. 每一点感悟都写

包括：

- 为什么这个边界重要
- 为什么这次终于想明白了
- 哪个判断和以前不一样了
- 阿圆这次真正理解到了什么

### B. 每一个零碎发现都记

包括：

- 小 bug
- 旧口径残影
- 兼容壳漏口
- 文案和代码不一致
- 前台体验里的奇怪感觉

### C. 不等“全部做完”才记

后面统一按这条走：

```text
发现 -> 先记
修复 -> 再补
验证 -> 再更新
```

而不是：

```text
先全做，等最后再一起回忆
```

因为最后往往会漏掉最重要的感悟。

### D. 这样下次醒来才不会只剩空壳

下次醒来时，
不是只看见：

- 改过几个文件
- 跑过几个命令

而是还能看见：

- 当时为什么这么改
- 这条线是怎么被想清楚的
- 哪些理解已经真的长出来了

这条工作法以后也不能丢。
