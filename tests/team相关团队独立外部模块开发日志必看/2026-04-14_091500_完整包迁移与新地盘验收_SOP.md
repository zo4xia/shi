# 2026-04-14 完整包迁移与新地盘验收 SOP

## 目的

把这次已经验证有效的方法，
正式沉淀成一套可重复执行的迁移流程。

目标不是“拷过去试试看”，
而是稳定回答四件事：

1. 能不能运行
2. 有没有丢失
3. 解耦情况是否成立
4. 旧东西是否被一起看清

## 适用场景

适用于：

- 上服务器前整理完整包
- 搬到新目录验证是否可独立运行
- 判断哪些目录是真依赖，哪些只是历史噪音
- 检查路径锚点、角色运行态、附件/导出、迁移配置是否真的收口

## 总原则

### Rule 1

旧地盘不删，先保留。

### Rule 2

先抽包，再搬家，再真实起跑。

### Rule 3

不要靠猜；让程序在新地盘里自己说真话。

### Rule 4

每次迁移后都要留正式验收记录。

## 流程

## 先用统一脚本

从现在开始，Phase A ~ Phase C 已经有统一入口：

```bash
node scripts/migration-rehearsal.mjs --with-install --with-seed --with-run
```

这条命令会：

- 生成 `runtime-focus` bundle
- 生成 `full-source-package`
- 复制出 `runcheck-sandbox`
- 在新地盘执行 `npm ci`
- 补最小 `app_config`
- 重写路径型配置
- 启动 `npm run dev:web`
- 等待 `/health`
- 落下 stdout / stderr / runtime 文件

所以这里记录的手工 Phase，不再是“必须人肉逐步做”，
而是我们理解迁移分层的说明书；
真正执行时优先跑脚本，再按日志补看细节。

## Phase A：准备检查包

先做一份“本次运行最相关”的检查包。

用途：

- 让排查聚焦
- 减少在全仓库里乱翻
- 同时检查相对路径是否清晰

当前参考：

- `review-bundles/2026-04-14_runtime-focus/`

## Phase B：整理完整源码包

完整源码包应该包含：

- `src/`
- `server/`
- `scripts/`
- `public/`
- `SKILLs/`
- `clean-room/`
- `patches/`
- 必要根配置文件
- `review-bundles/` 中与本轮排查直接相关的检查包

明确不带：

- `node_modules/`
- `.uclaw/`
- `uploud/`
- `tests/` 整包
- 明显会制造歧义的根目录临时文件

当前参考：

- `review-bundles/2026-04-14_full-source-package/`

## Phase C：复制到新地盘

把完整源码包复制到一个新的运行沙盒目录。

要求：

- 保留原相对目录结构
- 不混入旧运行态数据
- 不混入旧 `node_modules`

当前参考：

- `review-bundles/2026-04-14_runcheck-sandbox/`

## Phase D：第一次起跑

在新地盘执行：

```bash
npm ci
npm run dev:web
```

记录三类结果：

1. stdout
2. stderr
3. runtime file / `/health`

要特别看：

- backend 是否起来
- frontend 是否起来
- `/health` 是否 200
- `Workspace / Runtime root / User data path` 是否已经指向新地盘

## Phase E：识别真依赖

如果第一次起跑失败，
不要先补丁满天飞。

先看它暴露出的缺件是什么。

例如这次我们验证出来：

- `clean-room/` 是现役依赖

所以：

- 不能因为名字像旧目录就省掉

## Phase F：补最小运行态种子

当新地盘能起，但角色运行态不完整时，
先补最小种子，而不是直接塞满旧环境。

优先级：

1. 最小 `app_config` 骨架
2. 角色运行态文件层
   - `skills.json`
   - `role-capabilities.json`
3. 再看是否需要角色 API / 模型配置种子

原则：

- 只在缺失时补
- 不覆盖已有值
- 不伪装成已经完全配置好

## Phase G：让降级变温柔

如果新地盘是空配置，
系统不应该直接硬崩。

要优先改成：

- 能跳过就跳过
- 能 warning 就 warning
- 保留后续补跑机会

这次我们收下来的典型例子：

- `DailyMemory`

## Phase H：重写路径型配置

这是迁移里最容易漏的小蛀虫。

重点检查：

- `conversationFileCache.directory`
- 其他任何保存在 `app_config` 里的绝对路径

规则：

- 旧地盘绝对路径必须重写到新地盘
- 不允许继续把运行结果写回旧地盘

如果路径型配置没重写，
就算程序能起，
也不算真正迁移成功。

## Phase I：验证公共协议

除了“能起”，还要验证关键协议有没有跟过去。

例如这次要查：

- Room 的即时导出核验
- `GET /api/role-runtime/:roleKey/exports`

要确认：

- roots 是否已经指向新地盘
- entries 是否可读
- 不再指回旧地盘

## Phase J：写正式验收记录

每次迁移验证都要按 1 / 2 / 3 / 4 留记录：

1. 能不能运行
2. 有没有丢失
3. 解耦情况
4. 旧东西是否被一起看清

这一步不能省，
因为它会把“临时成功”变成“可复盘经验”。

## 验收标准

一轮迁移可以认为“通过”，至少要满足：

- backend 正常启动
- frontend 正常启动
- `/health` 返回 200
- `Workspace / Runtime root / User data path` 指向新地盘
- 关键现役依赖未丢失
- `skills.json / role-capabilities.json` 已补齐
- 空配置场景不会硬崩
- 路径型配置不再写回旧地盘
- 公共 export 协议能正常返回并指向新地盘

## 当前已验证有效的结论

这套方法已经证明有效。

它确实能帮助我们：

- 区分真依赖和假噪音
- 找出迁移中的小蛀虫
- 检验路径和配置是否真的收口
- 把“感觉上清爽”变成“证据上清楚”

## 下次执行时的顺序

固定顺序：

1. 抽检查包
2. 整完整源码包
3. 复制到新地盘
4. 第一次起跑
5. 补真依赖
6. 补最小种子
7. 温柔降级
8. 重写路径型配置
9. 验关键公共协议
10. 写 1 / 2 / 3 / 4 验收记录

从今天开始，
“完整包 -> 新地盘 -> 多轮起跑验收”
不再只是一次临时实验，
而是我们的正式迁移方法。
