## CoPaw 那边最值钱的地方

不是某一份 API 表本身，
而是他们做了三件事：

### 1. 给接口、数据、事件、实时更新分了不同符号

例如：

- `⚡` API 接入点
- `💾` 数据绑定
- `🔄` 实时更新
- `🔌` 事件监听
- `📡` WebSocket
- `🔐` 权限验证
- `⚠️` 错误处理
- `📦` 数据转换
- `🎨` 动态样式
- `🧩` 组件复用

### 2. 给每个点位都配了 ID

不是只写“这里要接接口”，
而是写成：

```text
ID: provider-list-get-001
```

这样以后不靠模糊记忆找。

### 3. 把设计稿和后端 API 一一对照

这很像施工图，
不是空讨论。

所以它们厉害的地方不是“文档多”，
而是：

**点位可找、链路可搜、问题可回钩。**

## 以后我们自己的混合打法

### 第一层：继续保留 `#路标`

因为 `#路标` 适合我们现在的连续性写法。

例如：

- `#路由_白板首页`
- `#业务流_TTS时间轴`
- `#边界_主家园保护`
- `#问题_独立运行环境`

这层像：

**整团毛线的大主题。**

### 第二层：在需要精确施工的位置，再加符号埋点

例如以后可以写成：

```text
#接口_语音_tts_timeline
⚡ API: generate-tts-timeline
💾 数据：audio.url / sentences.words
🔌 事件：点击“生成时间轴”
⚠️ 风险：3003 不在线时按钮链不推进
```

或者：

```text
#问题_播放
⚡ API: /api/tts-audio
📦 数据转换：远程 URL -> 本地代理 -> blob URL
⚠️ 风险：主页面与最小试验台双口径
```

也就是说：

- `#路标` 负责告诉我们这团线是什么
- `⚡💾🔄🔌...` 负责告诉我们这根线具体缠在哪

---

## 适合立刻开始用的符号

先不要一口气全上。

我们现在最适合先用这 6 个：

### `⚡`

用在：

- API 接入点
- Socket 事件入口
- 代理接口入口

### `💾`

用在：

- localStorage
- 页面状态绑定
- 返回数据结构

### `🔌`

用在：

- 按钮点击
- onSubmit
- onMount
- 触发 socket emit

### `⚠️`

用在：

- 当前已知问题
- 双口径
- 容易误判的地方
- 客户演示风险

### `📦`

用在：

- 数据结构映射
- `BoardInsertPoint -> HandwriteCraft`
- 远程音频 -> 本地代理

### `🎨`

用在：

- 页面状态变化
- 白板视觉状态
- 客户演示页的动态反馈

## 一句话工作法

以后发现一个问题，
不要只写：

```text
这个地方怪怪的
```

而是尽量写成：

```text
#问题_分类
⚡ API / 事件入口
💾 绑定的数据
🔌 触发动作
⚠️ 当前风险
📦 是否有结构映射
```

这样以后回来不是重新猜，
而是顺着毛线头一拉就能回到现场。

---

## 一句话收束

`#路标` 是我们的线团名字。  
`⚡💾🔄🔌⚠️📦🎨` 是线头上的小标签。  

以后大家一起学这个办法，
找问题会快很多。
