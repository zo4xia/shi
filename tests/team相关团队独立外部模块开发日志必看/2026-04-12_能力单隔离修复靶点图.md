# 2026-04-12 能力单隔离修复靶点图

## 结论先写在前面

这次要救的不是“整个 executor 都坏了”，而是要把“角色能力单”从“仓库技能扫描”里剥离出来。

按 `memory-safety-patterns` 的思路翻译：

- `roles/<role>/skills.json` 才是当前角色真正 `own` 的能力。
- `skillManager.listSkills()` 只是仓库候选清单，最多算 `borrowed view`，不能提升成运行时真相。
- 启动链和能力快照链现在把 `borrowed view` 当成了 `owned capability`，这就是边界污染。

## 已确认的真实链路

### 1. 门牌 prompt 和执行器主链相对干净

`server/libs/httpSessionExecutor.ts`

- `buildRoleHomePrompt()` 会把 `roleRoot`、`skillsIndex`、`capabilitySnapshot`、`skill-configs`、`skill-secrets`、`role notes`、`pitfalls`、`sqlite`、`attachment/export home` 明确告诉 agent。
- `resolveMergedSkillIds()` 当前只合并：
  - 角色运行时 `skills.json`
  - 本轮显式 `skillIds`

这说明执行器主链本身没有直接扫仓库技能目录。

### 2. 真正的主伤口在能力快照链

`server/libs/roleRuntimeViews.ts`

- `syncRoleCapabilitySnapshot()` 里先读角色索引，得到：
  - `availableSkills`
  - `globalAvailableSkills`
  - `roleBoundSkills`
- 但随后又调用 `skillManager.listSkills()`，额外生成：
  - `installedSkills`
  - `invalidBindings`
  - `unboundWorkspaceSkills`
- 最后这些仓库扫描结果被一并写进 `role-capabilities.json`

这就让 agent 看见的“能力单”不再纯粹，而是混入了仓库候选层。

### 3. 污染源头是 `listSkills()`

`src/main/skillManager.ts`

- `listSkills()` 会遍历多个 skills root
- 对每个技能目录执行 `parseSkillDir()`
- `parseSkillDir()` 会直接读取 `SKILL.md` 并解析 frontmatter

所以只要启动链或快照链碰 `listSkills()`，仓库里任何不干净的 `SKILL.md` 都可能把系统链路拖脏。

## 用“内存安全”语言重述问题

### 当前错误模型

- 把仓库技能目录当成“当前角色已拥有能力”
- 把候选技能元数据写进运行时门牌
- 让启动链持有本不该持有的外部状态

这相当于：

- 没有区分 ownership 和 borrowed reference
- 把不稳定外部资源直接放进长期结构体
- 让初始化路径持有过宽依赖

### 正确模型

- 角色能力真相：`roles/<role>/skills.json`
- 仓库技能清单：只用于绑定、排查、后台维护
- 运行时 prompt / capability snapshot：只展示当前角色实际已生效能力

## 修复优先级

### P0：先切断能力快照链对仓库扫描的依赖

第一修复位：

- `server/libs/roleRuntimeViews.ts`

目标：

- `role-capabilities.json` 默认只写入角色绑定能力和 runtime MCP / native capabilities
- 不再默认调用 `skillManager.listSkills()`
- `unboundWorkspaceSkills`、`invalidBindings` 这类“仓库体检项”移出主快照，改成独立诊断输出或显式诊断模式

完成标准：

- agent 门牌里的“能力单”不再受仓库 `SKILL.md` 解析结果污染
- 仓库脏 frontmatter 不再影响默认启动/快照链

### P1：保留 `listSkills()`，但把它降回仓库 API

第二修复位：

- `src/main/skillManager.ts`

目标：

- 明确注释和调用约束：`listSkills()` 只回答“仓库里有哪些技能候选”
- 不把它当作当前角色能力来源
- 后续如果要继续收紧，可以再拆：
  - `listWarehouseSkills()`
  - `listRoleBoundSkills()` 由角色索引单独提供

完成标准：

- 调用语义清楚
- 后续代码审查时能一眼看出“仓库视图”和“运行时真相”不是同一层

### P2：把启动链改成轻启动，重诊断后置

第三修复位：

- `server/src/index.ts`

目标：

- 启动时只加载最低限度基础能力
- 技能仓库扫描、角色体检、仓库一致性诊断改成：
  - 后置
  - 异步
  - 局部失败不拖垮主启动

完成标准：

- 启动不再因为仓库技能候选层问题被拖慢或拖脏
- “用户可添加内容”失败时不会影响主运行链

## 现在先不要动的地方

这些地方当前相对干净，先不要混修：

- `server/libs/httpSessionExecutor.ts` 的主执行器技能合并逻辑
- `role_home_*` 附件/导出边界
- `attachmentRuntime` 的附件读取主逻辑
- `broadcast_board_write` / continuity 相关工具面

原因：

- 这些不是这次“能力单被仓库扫描污染”的直接伤口
- 现在扩修只会放大回归面

## 推荐修法

### 最安全的第一刀

在 `roleRuntimeViews.ts` 中把仓库扫描信息从默认快照里剥离：

1. 先去掉默认路径上的 `skillManager.listSkills()`
2. 快照只保留：
   - `availableSkills`
   - `roleBoundSkills`
   - `globalAvailableSkills`
   - `runtimeMcpTools`
   - `runtimeNativeCapabilities`
   - 只依赖角色索引可推导出的 warning
3. 把以下内容改为诊断专用：
   - `invalidBindings`
   - `unboundWorkspaceSkills`
   - `installedSkillIds`

### 第二刀再考虑

如果还要继续修：

1. 给仓库诊断单独出文件
2. 或者只在显式管理员操作时触发仓库体检

## 验证清单

修完至少验证这四件事：

1. 仓库里故意放一个脏 frontmatter 的 `SKILL.md`，主启动不应受影响。
2. `role-capabilities.json` 仍能正确展示角色绑定能力。
3. `httpSessionExecutor` 生成的 role home prompt 仍能读到正确 `capabilitySnapshot`。
4. 显式选中的 `skillIds` 仍能在执行器主链正常合并。

## 一句话收束

这次不是要“修所有技能系统”，而是先把边界修正：

- 仓库候选 ≠ 当前能力
- borrowed warehouse view ≠ owned runtime capability
- 启动链必须轻，诊断链必须后置

先把这条边界钉住，家才不会再被同一类问题反复拖坏。
