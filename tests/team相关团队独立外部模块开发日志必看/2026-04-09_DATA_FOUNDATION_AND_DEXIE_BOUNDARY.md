## 这条为什么要钉死

我们已经把方向想清楚了：

```text
家要稳稳的。
浏览器存储可以做口袋，不可以做房子。
```

所以这份文档只做一件事：

```text
把“什么是真相源，什么只是缓存，二期数据库边界到哪”钉死。
```

---

## 1. 一期地基

### 主真相源

- `SQLite`
- 默认文件：
  - `.uclaw/web/uclaw.sqlite`

### 为什么是它

- 跟浏览器壳分离
- 卸浏览器、清浏览器缓存，不会直接把家清空
- 聊天全文、记忆、会话、归属、存证都能稳稳落盘

### 当前明确口径

```text
我们负责真实源的记录与存证。
SQLite 是一期的家。
```

---

## 2. Dexie 的定位

### 能做什么

- 前端缓存命中层
- 热数据预热层
- 降低重复 API / 重复查库压力

### 适合交给 Dexie 的东西

- 会话列表缓存
- 会话预览缓存
- 当前会话最近 N 条消息镜像
- 广播板快照
- 摘要结果缓存
- 搜索结果缓存
- 附件清单索引缓存
- query result cache / turn cache 一类可重建结果

### 不准交给 Dexie 的东西

- 完整聊天记录唯一副本
- 用户长期记忆唯一副本
- 章节正文唯一副本
- 附件唯一副本
- 审计与存证唯一副本

### 一句话原则

```text
Dexie 负责减压，
不负责背书。
```

也就是：

```text
Dexie 可以帮我们更快，
不可以帮我们记住。
```

---

## 3. 正确分工

### SQLite / 远程 SQL

负责：

- 完整聊天记录
- 用户记忆
- 会话归属
- 角色归属
- 时间线
- 存证与恢复

### Dexie

负责：

- 命中
- 秒开
- 热缓存
- 重复查询减压

### 两者关系

不是竞争关系，
是主从关系：

```text
SQLite / 远程 SQL = 真相源
Dexie            = 影子缓存层
```

---

## 4. 缓存命中规则

前端正确读取顺序：

1. 先读 Dexie
2. 命中则先渲染
3. 后台再查 SQLite / 服务端校验
4. 如有冲突，以真相源覆盖并回写 Dexie

### 必须带的缓存字段

- `sessionId`
- `agentRoleKey`
- `updatedAt`
- `sourceRevision`
- `ttl`

### 这样做的意义

- 把重复重活踹出去
- 不让我们自己的服务端抗所有重复读取
- 但也不让缓存壳反过来污染真相源

---

## 5. 我们家的三层记忆策略

这条也一起钉死：

### 第一层：广播板

- 用途：短接力、当前任务、最近结论
- 特征：快、轻、短

### 第二层：数据库摘要 / 索引

- 用途：按时间、关键词、角色、章节、会话定位
- 特征：检索层，不是全文替身

### 第三层：原文与附件

- 用途：聊天全文、章节正文、上传附件、导出物
- 特征：需要精确上下文时回到这里

### 当前推荐查找顺序

```text
广播板
  -> 数据库摘要 / 索引
  -> 以时间、关键词、特定文字为线索
  -> 查对话原文 / 附件
```

---

## 6. 二期边界

### 二期要做什么

- 支持用户自填远程数据库

### 为什么这样做

- 我们不承担每个用户数据库的运维
- 高性能需求的用户可以自己扩容
- 仍然符合“数据归用户”的原则

### 推荐口径

- 默认：
  - 本地 SQLite
- 进阶：
  - 用户自填远程 `Postgres`
- 语义检索增强：
  - `pgvector` 或外部向量库

### 但要继续守住

```text
向量库不是家，
只是检索增强层。
```

## 7. 最后一句钢印

```text
我们负责真实源的记录与存证，
把重复重活交给 Dexie 做缓存命中，
把数据库选择权在二期交还给用户。
```
