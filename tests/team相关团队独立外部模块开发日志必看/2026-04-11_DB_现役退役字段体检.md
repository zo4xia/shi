## 2. 已确认的现役主真相

### A. 对话主链

- `cowork_sessions` = 会话头
- `cowork_messages` = 正文主真相

这条线当前是明确接电的。

### B. 记忆主链

- `user_memory_sources` = 来源/活跃性辅助表
- `identity_thread_24h` = 24 小时广播板

- `user_memories` 保“长期记忆”
- `identity_thread_24h` 保“最近 24 小时跨终端连续性”
- 两者不是同一张表，也不是替代关系

### C. 技能与工具主链

- `skill_role_configs` = 角色技能绑定真相
- `mcp_servers` = MCP 配置真相

### D. 定时任务主链

- `scheduled_tasks` = 任务定义真相
- `scheduled_task_runs` = 运行记录真相

### E. 配置主链

- `kv` = 全局配置/运行态键值真相
- `cowork_config` = 主家园 cowork 局部配置真相

- 现役字段

- 字段存在

### 4.4 `user_memory_sources.model_id`

## 5. 当前最需要记住的“对应不上”

### A. Web `kv` 写入和 schema 曾经不对应

这次体检时确认到：

- `server/sqliteStore.web.ts` 的 `kv` schema 有 `updated_at`
- 但 `set()` 之前没写这个列

这属于：

- 不是口径问题
- 而是 schema / 写入直接不对应

本轮已修。

### B. `mcp_servers` 跨端 schema 仍不完全对齐

当前现状：

- 旧 Electron 库可能仍有 `model_id`
- 当前新建 Electron schema 已不再创建它
- Web schema 一直没有它
- 现役 `mcpStore` 也不用

这说明：

- 这一列不是现役字段
- 而是 drift 残影
- 当前代码已新增启动迁移：
  - 旧 Electron 库如果还带 `model_id`
  - 会重建 `mcp_servers` 并丢弃这列

### C. `identity_thread_24h` 跨端 schema 也不完全对齐

当前现状：

- 旧 Web 库可能仍有 `last_message_id`
- 当前新建 Web schema 已不再创建它
- 主 helper 完全不用
- Electron schema 甚至没有这列

这说明：

- 它既不是现役字段
- 也不是稳定跨端合同字段
- 当前代码已新增启动迁移：
  - 旧 Web 库如果还带 `last_message_id`
  - 会重建 `identity_thread_24h` 并丢弃这列

### D. 旧 `*_identity` 索引名已经开始退出现役 schema

这次已经继续收口：

- 新建 schema 不再继续创建这些旧名字
- 启动时会先 `DROP INDEX IF EXISTS idx_*_identity`
- 再重建成：
  - `role_enabled`
  - `role_created`
  - `role_model_*`

这代表：

```text
“identity = role + model” 这种误导性索引命名，
已经开始从现役 schema 退出。
```

### E. `user_memory_sources` 来源元信息列已从现役链退场

这次已经继续收口：

- 新建 schema 不再继续创建：
  - `agent_role_key`
  - `model_id`
- 旧库如果还带着这两列，启动迁移会重建并丢弃
- 现役写路径也不再继续写它们

这代表：

```text
user_memory_sources 重新回到“来源关系辅助表”的单一职责，
不再硬塞角色/模型元信息。
```
