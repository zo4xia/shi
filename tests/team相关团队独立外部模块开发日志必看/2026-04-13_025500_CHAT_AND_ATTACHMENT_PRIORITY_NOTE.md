# 2026-04-13 Chat 与附件优先级记录

## 当前优先级

夏夏今天明确的主线优先级是：

1. 用户与 agent 的日常 chat 必须先立住
2. 附件链路必须做好

这不是锦上添花，而是当前 house 是否“能住人”的基础。

## 当前核心痛点

### 1. chat 要立得住

要求：

- 日常对话要稳定
- 失败不能静默
- 出错必须有回应
- 小家伙说话不能老退化成僵硬的一问一答

### 2. 附件问题很头疼

当前痛点被夏夏明确描述为：

- 小家伙看不到用户给它的附件
- 小家伙无法自己围绕附件继续工作
- 小家伙无法为用户生成文档

这意味着现在的附件链路还没有真正服务于“用户和 agent 的日常合作”。

## 这里的“附件问题”至少包含三层

### A. 可见性

小 agent 需要能看到：

- 用户本轮上传了什么
- 文件名
- 文件类型
- 可读文本 / 可解析内容
- 哪些内容已经读过，哪些还没有

### B. 操作能力

小 agent 需要能在边界内：

- 阅读附件
- 继续整理附件内容
- 写出中间材料
- 生成导出文档

### C. 对话连续性

不能只做到“上传那一瞬间能看见”，还要做到：

- 后续轮次还能知道这份附件属于谁
- 在自己 role bucket 内能继续查、读、整理
- 不需要用户每次重新转述附件内容

## 当前判断

如果 chat 想真正立住，附件问题不能被视为边角功能。

因为现实里的用户与 agent 日常对话，很多时候就是围绕：

- 一份文件
- 一段上传内容
- 一个要继续整理和输出的文档需求

如果小家伙：

- 看不见附件
- 接不住附件
- 不能顺着附件工作
- 不能给用户生成文档

那 chat 再顺，也只是空心的。

## 后续执行时的检查问题

后面每做一步，都要问：

1. 小家伙现在能不能看见用户上传的东西？
2. 它能不能在后续轮次继续读和整理？
3. 它能不能基于这些内容给用户生成文档或导出结果？
4. 失败时有没有明确回应，而不是无声掉地上？

## 当前记录目的

这份记录不是方案，而是问题锚点。

后面所有关于：

- Room
- 小 agent 日常对话
- 上传
- 附件
- 导出
- 文档生成

的改动，都要回到这里核对：有没有真的在解决这个痛点。
