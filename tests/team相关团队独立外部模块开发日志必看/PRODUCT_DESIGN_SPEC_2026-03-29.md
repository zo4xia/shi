## 1. 产品概述

UCLAW 是一套面向多角色协作场景的 AI 工作平台，核心目标是把对话、角色分工、记忆、任务调度与外部渠道接入收敛到同一套产品主链中。

它不是单一聊天窗口，也不是单一自动化工具，而是一个围绕“身份驱动协作”建立的工作台：

- 用户可在同一产品中使用不同 AI 角色完成组织、写作、设计、分析等任务。
- Web 与 Feishu 渠道可共享同一角色身份的连续性和记忆。
- 定时任务、Skills、MCP、记忆条目等能力可围绕角色统一编排。

---

## 2. 设计目标

### 2.1 目标一：用身份而不是模型组织能力

UCLAW 当前的核心设计不是“选一个模型开始聊天”，而是“选一个角色身份进入工作”。

这意味着：

- 角色是用户认知入口
- 模型只是角色背后的运行配置
- 记忆、技能、任务、渠道绑定都围绕角色组织

### 2.2 目标二：让多入口共享同一条工作主线

产品当前主线同时支持：

- Web 对话入口
- Feishu 渠道入口
- Scheduler 定时触发入口

无论从哪个入口进入，只要绑定到同一 `agentRoleKey`，就应共享同一套身份记忆与连续性。

### 2.3 目标三：让扩展能力在主链内可控生长

产品不是封闭系统，当前已支持：

- Skills
- MCP
- 文档解析
- 定时任务

设计上强调“可接入，但要收口”，避免扩展能力反向破坏主链稳定性。

## 3. 适用场景

### 3.1 内容生产与团队协作

- 多角色协同写作
- 内容策划与摘要整理
- 文案与设计辅助
- 研究分析与资料归纳

### 3.2 渠道协作

- Web 端发起与管理会话
- Feishu 中由指定角色承接消息与任务
- 跨渠道共享身份记忆

### 3.3 任务自动化

- 定时执行固定提示词任务
- 按角色运行日常整理、汇总、检查类流程
- 保存运行历史，便于追踪结果

## 4. 核心设计特点

### 4.1 角色优先

当前一期固定四角色：

- `organizer`
- `writer`
- `designer`
- `analyst`

产品将角色作为：

- 会话上下文入口
- 渠道绑定目标
- Skills / MCP 作用域
- 记忆归属边界

### 4.2 记忆连续性优先

产品内已存在：

- 24h 身份热线程
- 用户记忆条目
- 记忆来源链
- 每日记忆抽取

这意味着系统不是“每次重新开始”，而是能围绕角色身份维持连续工作上下文。

### 4.3 渠道统一而不混乱

当前主线明确支持 Web 与 Feishu。

设计原则不是把不同渠道当成独立产品，而是：

- 渠道只是入口
- 角色才是主线
- 渠道能力应服务于同一套身份连续性

### 4.4 扩展能力模块化

产品当前支持：

- Skills：用于增加特定工作能力
- MCP：用于挂接外部工具或服务
- 文档解析：处理常见附件与文本资源

设计上保持“核心链路稳定，扩展能力外挂式接入”。

- 续聊

- 会话历史
- 标题管理

- 角色身份隔离

- 支持围绕角色组织能力与记忆

### 5.3 Feishu 接入

- 接收飞书消息
- 绑定角色
- 将消息纳入同一身份连续性链路

### 5.4 定时任务

- 创建任务
- 编辑任务
- 启停任务
- 手动执行
- 查看运行历史

### 5.5 Skills / MCP

- 技能安装与管理
- MCP 服务接入与管理
- 角色绑定
- 市场/目录视图

- `md`
- `csv`
- `json`

## 6. 架构分层

### 6.1 前端层

当前前端采用：

- React
- TypeScript
- Vite
- Tailwind
- Redux Toolkit

负责：

- 页面与交互
- 会话视图
- 设置面板
- Skills / MCP 管理页
- 调度与记录视图

### 6.2 服务层

当前后端采用：

- Express
- WebSocket

负责：

- API 路由
- 会话执行链路
- 配置读写
- 渠道接入
- 调度任务
- 文件与运行态服务

### 6.3 数据层

当前产品使用：

- `sql.js` 持久化 SQLite 文件

默认运行数据目录收口在项目内：

- `./.uclaw/web`

### 6.4 兼容层

前端仍保留 `window.electron` 兼容外形，以承接历史代码并统一调用面。

这一层的价值在于：

- 维持前端调用稳定
- 逐步从旧形态过渡到现役 Web 主线

## 8. 产品优势总结

相较于“单聊天框 + 单模型”的产品形态，UCLAW 当前主线的优势在于：

- 以角色组织协作，而不是只以模型组织对话
- 让 Web、Feishu、Scheduler 共享同一工作主链
- 把记忆、技能、渠道、任务围绕身份统一编排
- 为后续扩展 Skills、MCP、文档处理保留结构化空间
