## 1. 当前最硬的刺点

### A. `kv.updated_at` 的 schema / 写入不对应

坏蛋视角结论：

- 这是会直接炸的真 bug
- 不是“可能有风险”

当时问题：

- Web `kv` schema 只有 `key/value`
- 但 `set()` 已开始写 `updated_at`
- 主进程里 `CoworkStore.setKvValue()` 也还在裸写两列

状态：

- 已当场修

### B. 旧库 rebuild 不是事务型

坏蛋视角结论：

- 如果 rename / create / insert 中途失败
- 现役表可能变空，旧数据躺在 `__legacy`

状态：

- 已当场修
- 至少主进程和 Web 的 `rebuildTableDroppingColumns()` 都补了事务包裹

### C. `identity_thread_24h` rebuild 会撞 `UNIQUE(agent_role_key)`

坏蛋视角结论：

- 如果旧库还残留同 role 多行
- 按原样 `INSERT ... SELECT` 回填会直接炸唯一键

状态：

- 已当场修
- Web rebuild 改成先按 `agent_role_key` 分组，只保每个 role 最新一行

代价：

- 这是“允许清旧不兼容数据”的收法
- 不是无损合并

### D. `last_used_at` 只在一条写链接电

坏蛋视角结论：

- `coworkStore` 已接电
- `identityMemoryManager` 还没跟上
- 同表里一部分记录有使用时间，一部分永远空

状态：

- 已当场修
- `identityMemoryManager` 也已补上 `last_used_at`
