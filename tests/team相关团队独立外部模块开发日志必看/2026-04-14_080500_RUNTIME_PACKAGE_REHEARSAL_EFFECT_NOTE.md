# 2026-04-14 抽完整包到新地盘运行验证：这招很见效

## 今天做了什么

我们没有继续在旧地盘里一边猜一边清，
而是做了两层动作：

1. 先抽出本次运行最相关的检查包
2. 再整理出一个完整源码包
3. 把完整包复制到新的运行沙盒里，真实启动一次

现在这三步已经能由同一条命令重放：

```bash
node scripts/migration-rehearsal.mjs --with-install --with-seed --with-run
```

也就是说，
“检查包 -> 完整包 -> 新地盘” 这条被证明有效的动作链，
已经不再需要靠手工复制粘贴维持。

## 为什么这招有效

因为它不再让我们停留在“感觉上应该没问题”。

它会直接逼出四类真相：

1. **能不能运行**
2. **有没有丢件**
3. **解耦到底做到了哪一步**
4. **旧东西哪些其实还是现役依赖**

这四件事，
在旧地盘里很容易被噪音淹没。

但一旦搬到新的地盘，
问题会变得非常诚实。

## 这次具体看清了什么

### 1. 路径锚点是真的成立了

新地盘启动后：

- `Workspace`
- `Runtime root`
- `User data path`

都跟着新地盘走了。

这说明这段时间收的：

- `projectRoot`
- `runtimeRoot`
- `userDataPath`

这条主线不是纸上谈兵，
而是真的能带着程序一起迁移。

### 2. `clean-room/` 不是可随手丢掉的旧目录

第一次起跑失败时，
直接报：

```text
Cannot find module '../../clean-room/spine/modules/inbound'
```

也就是说：

- `clean-room/` 现在仍然是现役依赖
- 它不能因为名字像“旧实验目录”就被误删

这类问题在旧地盘里很容易被忽略，
但在新地盘里会立刻暴露。

### 3. 角色运行态文件层还没有自动种好

新地盘启动后，
服务端虽然起来了，
但会提示：

- `skills.json` 缺失
- `role-capabilities.json` 缺失

这说明：

- 源码和入口已经能独立跑
- 但“角色运行态初始化”还不是自动闭环

这不是坏事，
因为现在问题被看见了。

### 4. `DailyMemory` 的降级还不够温柔

新地盘没有角色配置时，
`DailyMemory` 会报：

```text
没有可用的 Agent Role 配置，无法调用 LLM 进行摘要
```

这说明：

- 它的主逻辑没错
- 但在“新地盘 / 空配置 / 首次迁移”场景下，
  它还不够温柔，
  还需要一个“无配置就跳过”的降级分支

## 这次实践带来的经验

以后遇到：

- 仓库太杂
- 主线和旧壳混在一起
- 怀疑有残留污染
- 想确认一个包是不是能独立运行

不要只在原地反复猜。

更有效的方法是：

1. 先抽“本次运行相关检查包”
2. 再做“完整源码包”
3. 复制到新地盘真实运行
4. 用运行结果倒逼我们认清：
   - 真依赖
   - 假噪音
   - 缺的初始化层
   - 不够温柔的降级逻辑

## 这招为什么适合我们

因为我们现在做的系统很容易遇到：

- 角色运行态
- 路径锚点
- 工具边界
- skills / runtime / cache / memory 多层交错

如果只在旧地盘清理，
人脑很容易被历史残留误导。

但一旦复制到新地盘，
程序会帮我们说真话。

## 当前结论

这次“抽完整包 -> 新地盘运行”这招，
不是折腾，
而是高效的验证方法。

它确实让我们一下子清爽很多。

因为：

- 主线更清楚了
- 真依赖暴露出来了
- 假噪音也开始和主线分开了

这条经验值得留下。

并且现在已经有了对应的统一脚本入口，
以后要重演“哪一层会先暴露缺件、哪一层会先暴露暖机行为”，
可以直接重复跑同一套 rehearsal，而不是重新手工摆一次阵。
