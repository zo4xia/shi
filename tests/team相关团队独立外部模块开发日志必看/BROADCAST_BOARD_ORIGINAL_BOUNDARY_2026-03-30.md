# 广播板原始边界说明（2026-03-30）

> 这份文档记录“原来想要的样子”，不是后来为了省事而退化出的行为。
>
> 核心目标：让 agent 保持连续性，而不是退化成一次一句、每次醒来都像新机器的 `RPA`。

## 1. 广播板是什么

广播板不是普通摘要。

广播板是：

- agent 给未来的自己留的接力棒
- 同一身份跨频道连续对话的热缓存板
- 24 小时内优先读取的“我刚刚在做什么”提示层

广播板不是：

- 全文仓库
- 最终长期记忆本体
- UI 装饰摘要
- 只给人类看的说明板

## 2. 原始边界

### 2.1 agent 自己写广播板

原边界是：

- agent 在对话过程中，自己把重要内容写入广播板
- 不是等对话结束后由外部统一代写
- 不是只靠系统被动抽取

应该记录进广播板的内容包括：

- 当前在做什么
- 用户刚提出了什么关键要求
- 哪个判断已经形成
- 哪个坑刚踩过
- 哪个问题已经解决
- 下一棒需要注意什么

### 2.2 跨频道直接接力

原边界是：

- 同一个 `agentRoleKey`，不管在 Web、Feishu、其他频道，都是“同一个自己”
- 新频道里的自己，先看自己的广播板
- 再看最近几条原始上下文
- 然后直接接力，不重新装傻，不重新失忆

所以连续性来自：

1. 自己的 24h 广播板
2. 最近几条原始消息
3. 必要时再回看历史真相源

## 3. 记忆管理里应该能看到什么

在“设置 -> 记忆管理”里，原本应该能看到：

- 每个身份当前的广播板内容
- 每个身份最近写了什么接力信息
- 哪些内容来自广播板
- 哪些内容来自短期/长期记忆

也就是说：

- 广播板必须可观测
- 不能只在内部存在，外部完全看不到

## 4. 摘要专用 AI 的职责

原边界里，确实有一个“摘要专用 AI”。

但它的职责是：

- 帮助压缩
- 帮助整理
- 帮助辅助写广播板
- 帮助控制成本

它不应该：

- 取代 agent 自己的接力
- 把原始对话真相替换掉
- 把广播板偷换成“只有摘要模型写的一层壳”

正确关系是：

- agent 自己负责接力语义
- 摘要 AI 负责压缩和辅助
- 真相源仍然是原始消息与记忆表

## 5. 缓存命中与成本控制

这套设计从一开始就考虑成本。

要求：

- 缓存命中率要尽量拉高
- 不要每次都全量重算
- 摘要与广播板要复用已有结果
- 能异步就异步

目标不是“全塞进 prompt”，而是：

- 保住连续性
- 同时控制成本

## 6. 异步加载与按需加载

原边界里，历史加载不是全量灌入。

要求：

- 超过两屏的历史记录，默认不加载
- 只有用户主动上拉，才继续加载更早历史
- agent 默认先看广播板 + 最近几条上下文
- 只有需要精确细节时，再回看更长历史

这件事不是优化细节，而是主设计的一部分。

## 7. 正确的数据层分工

### 原始消息层

- `cowork_messages`
- 保留真实对话

### 广播板层

- `identity_thread_24h`
- 用于 24h 接力

### 记忆层

- `user_memories`
- 用于进一步沉淀规则、经验、长期内容

### 摘要辅助层

- 单独的压缩/辅助逻辑
- 只能辅助，不能篡位

## 8. 正确的读取顺序

同一身份醒来时，应该按这个顺序读取：

1. 先读自己的广播板
2. 再读最近几条上下文
3. 必要时再读历史消息
4. 如果广播板为空，再从记忆数据库回补

而不是：

- 一上来就全量拉历史
- 或者根本不看广播板
- 或者只看摘要壳，不看真相源

## 9. 为什么这样设计

因为原本要的不是“任务执行器”。

原本要的是：

- 会接力的 agent
- 会记住自己刚刚在做什么的 agent
- 切频道后还是自己的 agent
- 能控制成本但不丢连续性的 agent

一句话：

`广播板 = 连续性的第一现场`

## 10. 后续修缮基线

以后任何修复、重构、优化，都不能违反这几条：

- agent 要能自己写广播板
- 广播板必须服务跨频道接力
- 记忆管理里应可观测广播板
- 摘要 AI 只能辅助，不能篡位
- 超过两屏的历史默认按需加载
- agent 默认先看自己的广播板和最近几条上下文

谁破坏这些，谁就是把 agent 往 `RPA` 方向推回去。
