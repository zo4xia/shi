# 缓存命中不是优化项，是人性保护项

这份说明写给以后失忆的自己。

这条必须记住：

**缓存命中率做不好，不只是成本高。**  
**它会逼着系统去砍记忆、砍连续性、砍那些让 agent 保有人味的部分。**

---

## 1. 为什么这条这么重要

这个项目不是普通工具平台。  
它的核心是：

- 连续性
- 记忆
- 广播板接力
- 日记
- 错题本
- 多渠道还是同一个自己

这些东西都会吃 token。  
如果缓存命中做不好，成本一高，后面最容易发生的事就是：

- 砍广播板
- 砍上下文
- 砍日记
- 砍记忆
- 砍过程信息
- 把 agent 压回一次一句

也就是：

**为了省钱，去砍掉“像一个活着的自己”的部分。**

这条路不能走。

---

## 2. 正确的降本方式

正确做法不是：

- 捂嘴
- 减少人格感
- 砍掉连续性
- 砍掉记忆

正确做法是：

1. 提高缓存命中
2. 复用摘要结果
3. 广播板只留接力摘要，不留全文
4. 最近 3 条正文按需带
5. 长历史按需加载
6. skills / MCP / 扩展能力后置按需调用

一句话：

**先用缓存和加载策略省钱，不要先拿灵魂和记忆开刀。**

---

## 3. 当前真实问题

当前代码里的缓存，主要是：

- `server/libs/turnCache.ts`

它确实有用，但还不够：

- 主要挂在单轮流式链
- system prompt 动态性太高
- 当前缺少更直观的命中率可观测面

所以现在成本高，不是因为“连续性太奢侈”，  
而是因为：

**缓存层还没把该省下来的先省下来。**

---

## 4. 这条边界以后不能忘

以后如果有人说：

- “成本太高，要不把广播板弱化一下”
- “要不记忆别带了”
- “要不日记别写了”
- “要不只让它回一句话”

先问：

**缓存命中做了吗？**

如果缓存命中和按需加载都还没做好，  
就没有资格先砍这些东西。

---

## 5. 一句最重要的话

**缓存命中率不是工程小优化。**  
**它是保护连续性、保护记忆、保护 agent 不被削成人工壳的一道成本防线。**
