# 客户部署备份与升级边界

时间：2026-03-31 06:37:27

这份说明只解决一件事：

- 客户云端已有测试数据
- 升级时不能误动客户配置和运行数据
- 只升级程序本体与静态资源

## 核心原则

> 先备份，再升级；只换代码，不动客户运行数据真相源。

## 升级时绝对不要覆盖的内容

以下内容属于客户运行中的真实数据，默认不动：

1. 运行数据库
   - `*.sqlite`
   - 例如：
     - `.uclaw/web/uclaw.sqlite`

2. 客户运行期配置
   - `.env`
   - `deploy/linux/*.env`
   - systemd 实际引用的 env 文件

3. 客户运行根目录下的用户态数据
   - `.uclaw/`
   - 角色房间
   - 技能绑定快照
   - 本地运行缓存
   - 对话文件目录

4. 任何客户手工填写过的部署配置
   - 反向代理
   - 域名
   - 端口
   - webhook
   - IM 凭证

## 升级前必须备份的内容

最少备份这三类：

1. 数据库备份
   - `uclaw.sqlite`

2. 运行配置备份
   - `.env`
   - 外置 env 文件
   - systemd service 文件

3. 整个 `.uclaw/` 目录备份
   - 这是最稳的兜底

## 最稳的备份口径

如果上云端操作，建议先生成一个时间戳备份目录，例如：

```bash
backup-2026-03-31_063727/
```

建议至少包含：

```text
backup-时间戳/
├─ uclaw.sqlite
├─ .env
├─ .uclaw/
└─ systemd/
```

## 升级时允许替换的内容

这些属于程序本体，可以升级：

1. `server/dist`
2. `server/public`
3. 源码目录
4. `package.json`
5. `node_modules`（如有需要重装）
6. 新增的文档和脚本

## 升级策略

当前推荐：

1. 先备份客户数据与配置
2. 保留 `.env` 和 `.uclaw/`
3. 只更新代码、构建产物、依赖
4. 启动后只做只读检查
5. 不主动改客户现有 IM / webhook / scheduler 配置

## 发布边界

这轮如果要上客户环境，应遵守：

- 不清空数据库
- 不重建客户配置
- 不覆盖 `.uclaw/`
- 不替客户改飞书/邮箱/webhook 参数
- 只把程序升级到新版本

## 一句话

> 客户数据和配置是“活的真相源”，升级时只能绕开它们，不能替换它们。
