# 2026-04-02 最小稳定底座与复用边界

## 背景

当前项目进入白天高压使用阶段。

这时的原则不是继续造能力，不是继续铺大而全，而是先守住用户每天真实会点、点了不能炸的基础链路。

同时明确：

- 不重复造轮子
- 优先复用成熟组件与成熟能力
- 把珍贵精力花在真正值得守护的东西上

## 当前总原则

### 1. 先做减法，不做扩张

现阶段不追求功能越多越好，先追求：

- 基础使用稳定
- 云端/本地不轻易卡死
- 关键导出链路可用
- 用户看得懂系统是否还在工作

### 2. 不造轮子，优先复用

后续方向明确为：

- 优先复用成熟 UI/能力底座
- 能用组件就不用一层层临时样式堆叠
- 能借成熟生态就不自己重复发明

方向上认可 LobeHub 生态适合作为后续复用来源，但当前阶段只考虑“按需借组件”，不做整套系统迁移。

### 3. 底子要能像乐高积木一样拼

以后项目里的能力与能力之间，需要能像积木一样组合，而不是继续长成一整坨互相缠绕的页面逻辑。

因此后续实现原则：

- 先组件化
- 再页面化
- 先能力块
- 再皮肤层

## 当前明确不做

以下内容当前阶段先不推进或暂缓：

- TTS
- 大规模新能力扩张
- 整套 UI 框架迁移
- 非必要的复杂前端重构

原因：

- 现在要保稳定
- TTS 不是当前最小可用链路
- 客户已经在用，优先避免新增不稳定因素

## 当前最小稳定目标

这几条是现阶段必须优先守住的基础链路：

1. 对话发送稳定
2. 长任务不要“假死”，至少让用户看得见在做事
3. 文本导出可用
4. 图片导出/图片保存可用
5. 禅模式与广播板边界清楚
6. 设置页中与运行链直接相关的部分稳定

## 当前产品边界决策

### 禅模式

禅模式作为用户主动开关，不做任务类型猜测。

禅模式下：

- 关闭广播板读写
- 不读取广播板连续性
- 保留工具调用
- 保留 skills
- 保留数据库记忆查询
- 保留正常会话能力

这是一种类似“无痕/隔离思路”的轻量运行边界，而不是删功能。

### 超时策略

不能再把短超时当成统一产品边界。

长文创作、爬虫、复杂工具链任务，重点应该是：

- 让用户看到系统仍在执行
- 展示思考/工具调用状态
- 允许用户手动打断

超时更适合作为底层保护兜底，而不是过早砍掉任务。

## 当前执行优先级

按稳定性优先，当前建议顺序：

1. 修云端导出图片/保存图片
2. 修导出对话记录
3. 补“清空当前广播板”按钮
4. 强化长任务状态可见性

## 一句话结论

先保命，再收纳，最后再升级。

现在最重要的不是做更多，而是把最小基础用户链路稳稳守住。
