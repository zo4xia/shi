# 工程执行宪法（2026-03-30）

> 用途：把“怎么做事”写死到仓库里，避免每次重启、换人、换上下文后又回到无序摸黑。
>
> 这不是某个外部项目的实现复刻，而是当前仓库的工程工作法。

## 1. 先记住的事

- 先读规则，再动代码。
- 先找真相源，再看表现层。
- 先钉主链，再修边角。
- 先埋点和标记，再谈实现。
- 先全局搜同类问题，再宣布修复完成。
- 允许 UI 降噪，不允许真相源失真。
- 每次踩坑、调通、失败、修复，都要写入错题集。

## 2. 真相源优先级

排查与实现时，优先级固定如下：

1. SQLite / 文件系统运行时真相源
2. store / route / executor / finalizer 主链
3. service / state / websocket 同步层
4. renderer 页面与组件
5. 文案、样式、装饰性表现

禁止倒过来做。

## 3. 核心工作顺序

### 步骤 1：预载规则

开始前先读这些材料：

- `docs/AGENTS.md`
- `docs/DB_API_TRUNK_WALK_2026-03-27.md`
- `docs/PAGE_SERVICE_ROUTE_MAP_2026-03-27.md`
- 相关模块自己的设计/说明文档

如果还有外部相似项目的经验文档，只吸收方法，不搬实现。

### 步骤 2：Recon 侦查

至少完成这些侦查动作：

- 看根目录结构与入口文件
- 看数据库表与关键字段
- 看 route 挂载顺序
- 看请求 context 注入内容
- 看当前现役执行器是谁
- 看最终落库、落记忆、落广播板的收尾点

### 步骤 3：钉主链

每次必须回答清楚：

- 真实入口在哪
- 真实持久化在哪
- 真实执行器是谁
- 真实记忆写入在哪
- 哪些是 display-only
- 哪些是 compatibility shell

### 步骤 4：再做修复

修复顺序固定：

1. 主链断点
2. 数据失真
3. 身份越界
4. 上下文压扁
5. UI 错位 / 样式问题

## 4. 埋点与标记规则

核心代码、核心路由、关键转换点，必须显式打标签。

推荐统一使用：

- `⚡` API 接入点
- `💾` 数据绑定 / 持久化点
- `🔌` 事件触发点
- `🔄` 实时更新 / 订阅 / 重试
- `📡` WebSocket / 长连接
- `🔐` 权限验证
- `⚠️` 错误处理
- `📦` 数据转换
- `🎨` 动态样式
- `🧩` 组件复用

代码内继续沿用仓库已有 grep 友好标签：

- `{标记}`
- `{ROUTE}`
- `{API}`
- `{FLOW}`
- `{BREAKPOINT}`
- `{PROGRESS}`

要求：

- 核心路由必须有 `{FLOW}` 或 `{ROUTE}`
- 关键副作用点必须有 `{标记}`
- 容易回归的地方必须有 `{BREAKPOINT}`
- 做完未回写标签，视为没做完

## 5. 错误处理纪律

发现一个问题时，不允许只修眼前这一个点。

必须补做：

- 全局搜索同类写法
- 判断是单点 bug、模式性 bug，还是架构性失真
- 记录复现条件
- 记录解决方法
- 记录以后如何更快定位

原则：

- 已知错误不能第二次以同样方式出现
- 不要一边捡芝麻一边漏西瓜
- 编译、编码、类型、结构性问题要尽量斩草除根

## 6. 记忆与错题集纪律

工具调用、排障、接口兼容、环境问题、踩坑与修复，都属于必须沉淀的记忆。

至少要记录：

- 错误现象
- 触发条件
- 根因判断
- 修复动作
- 关联文件 / 路由 / 表字段
- 是否还有同类风险点

推荐把这些内容沉淀到：

- 运行时记忆系统
- 专门的工程错题集文档
- 路由/模块旁的标签注释

目标不是“记日志”，而是给未来的自己留接力棒。

## 6.1 禁止无证据推翻已验证修复

如果某个方案已经被用户在真实页面、真实渠道、真实演示场景中验证过：

- 不允许因为猜测就回退
- 不允许因为重启就默认覆盖
- 不允许因为“看起来更整洁”就替换掉已验证方案

必须先做：

- 对照当前验证结果
- 明确说明为什么要改
- 明确列出风险
- 明确保留回退路径

没有证据，不得推翻。
没有比对，不得重写。
没有记录，不得宣称“优化”。

## 7. 真相与展示的边界

必须严格区分两类东西：

### 真相源

- `cowork_messages`
- `user_memories`
- `identity_thread_24h`
- `kv(app_config / im_config)`
- 执行器原始事件流

### 展示层

- 消息折叠
- UI 去噪
- 省略展示
- 卡片摘要
- 导航聚合

规则：

- 展示层可以折叠
- 真相源不能压扁
- 摘要不能反过来覆盖原始记录
- 记忆抽取必须基于真实对话，而不是基于被压缩后的 UI 文本

## 8. 身份与上下文纪律

当前项目必须继续遵守：

- `agentRoleKey` 是身份真理
- `modelId` 不是身份
- 上下文必须异步、分层加载
- 记忆、任务、skills、MCP、24h 热缓存都按身份隔离
- 不允许为了“省事”把多段真实回复合并成一句

红线：

1. 不捂嘴
2. 不串脑
3. 不把摘要当原文
4. 不把兼容壳误判成主干

## 9. 文档纪律

Mermaid 图、时序图、路由图、架构图，不是装饰品，是指南针。

做功能或修主链时，优先维护这些产物：

- 数据流图
- 业务时序图
- 页面到服务到路由映射
- 关键 API 和数据库对应关系

要求：

- 文档目录要精简
- 临时测试散稿及时清理
- 过时文档不要污染主认知
- 主线文档必须可检索、可追踪、可回放

## 10. 环境与可移植性

禁止把个人机器环境写死进工程逻辑。

必须遵守：

- 不依赖绝对本地路径
- 环境变量只做部署兜底，不做默认真相源
- 优先运行态目录与数据库配置
- 不把 secret 混入前端可见配置

## 11. 页面分析标准动作

分析一个页面时，至少要回答：

- 页面由哪些组件组成
- 数据从哪里来
- 页面触发哪些 API
- API 对应哪个 route
- route 落到哪个 store / executor / 表
- 哪些状态来自 websocket / 轮询 / 本地缓存
- 哪些地方有错误处理 / 权限 / 转换 / 动态样式

最后最好补一份 SOP 或 Mermaid。

## 12. 执行结语

接手工程时，不要只当执行者，要切到领导者视角：

- 定期复盘
- 快速处理小债务
- 问题和任务建立映射
- 类型安全当架构，不当语法
- 文档要自解释
- 核心路由和断点必须可追踪

如果没有留下标签、文档、错题、链路图，只留下了“我改过”，那等于没有真正完成。
