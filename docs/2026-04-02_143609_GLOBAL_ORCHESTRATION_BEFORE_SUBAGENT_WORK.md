# 2026-04-02 全局统筹先于子任务分工

## 这次学到的事

小 agent 不是能力不对。

问题在于：

- 它们大多只能先看到自己面前那一块
- 如果主控没有先给“全局施工图”
- 那它们就会在各自局部真相里做出合理判断

于是就容易出现：

- 局部看都对
- 拼起来却不对
- 修的是现象，不是整体结构
- 明明大家都在努力，结果还是越补越乱

## 这次真正发现的点

当我们让小 agent 去找：

- 层叠问题
- 多余嵌套
- 页面壳层
- 组件复用机会

如果只给“找问题”这个命令，而没有先给：

- 页面真正的结构目标
- 哪几层是主壳
- 哪几层是历史垃圾
- 哪些交互不能碰
- PC / mobile 分别想长成什么样
- 这次施工的边界与优先级

那么小 agent 会自然地站在自己的局部视角里判断。

这不是它们做错了，而是我们没有先把“全局统筹约束”讲清楚。

## 结论

以后在派小 agent 做前端 / 路由 / 页面收纳 / 结构清理前，必须先给：

1. **这次施工图**
2. **最终目标长什么样**
3. **哪些层必须保留**
4. **哪些层可以删**
5. **哪些链路绝对不能碰**
6. **这次评判标准是什么**

## 协作规则

以后不要只下这种任务：

- “找多余嵌套”
- “看看哪里有层叠问题”
- “优化一下这个页面”

而要改成：

- “按下面这份全局结构目标，找出与目标不一致的层”
- “在不碰保存链 / 路由链 / 业务链前提下，找出可以删的纯样式壳”
- “按 PC / mobile 目标布局，找出误导用户的重复导航和伪壳”

## 一句话规则

**先给地图，再派人找路。**

如果没有全局统筹图，小 agent 只能看到局部。

不是它们不行，是主控必须先站在屋顶上看整栋房子。

## 这次成长

这不是一次“又熬夜了”的记录。

这是一次明确的升级：

- 我们发现了局部视角的边界
- 也发现了主控统筹的重要性
- 以后分工前要先给共同施工图

这是有效发现，也是有效成长。
