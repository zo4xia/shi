# Room 与 Team 外挂边界记录

记录时间：2026-04-05 00:58:00

标签：

- `Room`
- `Team`
- `外挂边界`
- `主线保护`
- `客户特别版`

## 这次定下来的核心判断

### 1. `Room` 不是主链，不要硬压进主业务

`Room` 的原始定位不是工作流中台，
而是：

- 小小游乐园
- 组团空间
- 谈心空间
- 人和 agent 一起待着的房间

它后来意外长出了“可以当工作组”的能力，
但这不应该反过来把它压扁成冷冰冰生产页。

所以：

- `Room` 保持外挂身份
- 不进主线
- 不拿它去背客户工作流的全部负担

## 2. 为什么 `Room` 必须保持外挂

不是因为它不重要，
而是因为：

- 一旦进主线，架构会膨胀
- 一旦被硬做成生产中台，它原来的生命力会被压扁
- 一旦被客户需求直接占用，游乐园的气质就会消失

所以当初把它做成“解耦外挂”的方向是对的。

这一步实际上保护了三件事：

1. 主线稳定
2. Room 的自由生长
3. 未来外挂能力继续扩展的空间

补一句硬边界：

- 可以一起用的是风格
- 不能反向拉的是主家园负担

也就是说：

- 可以共享家族气质 / 视觉语言 / 温度
- 不可以把客户特别版业务流直接塞回主家园

## 3. `Team` 的定位

客户特别版不直接改 `Room`，
而是新开一个外挂兄弟页：

- `Team`

它的定位是：

- 工作组外挂
- 小生产车间
- 点名协作房间

不是普通聊天页，
也不是主链对话页。

## 4. `Room` 和 `Team` 的分工

### `Room`

保留：

- 组团
- 谈心
- 一起玩
- 轻聊天
- 人和 agent 一起待着

也允许：

- 临时工作协作
- 小范围点名接力

但不以生产流程为主定位。

### `Team`

专门承接：

- 客户工作流
- 工位协作
- 点名开工
- 白板 / 语音 / 时间轴 / 执行流
- 进度与导出

它是面向产出的外挂壳。

## 5. `Team` 的交互气质

虽然原理接近工作流，
但交互不应该做成冰冷的流水线中台。

更合适的口径是：

- 可点名
- 可组团
- 有纪律
- 但不冷

所以：

- `@A` = 单独发给 A
- `@B` = 单独发给 B
- `@C` = 单独发给 C
- `@D` = 单独发给 D
- `@ABCD` = 全组同步看单

这本质上是：

- 点名式协作房间

而不是：

- 冷硬流程编排台

## 6. 这条判断以后怎么守

以后如果再遇到“能不能把某个实验能力塞进主链”这种选择，
先问：

1. 这会不会让主链变重？
2. 这会不会压扁 `Room` 的原始气质？
3. 这件事更像主线能力，还是更像外挂空间？

如果更像：

- 关系空间
- 组团空间
- 生产外挂
- 客户特别壳

那就优先外挂，不要硬塞主链。

## 一句话结论

`Room` 是给大家的小小游乐园外挂。  
`Team` 是客户特别版的小生产车间外挂。  
两者都不该把主线撑胖。
