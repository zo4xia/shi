# 连续性非回退告示（写给未来接手的 agent / 维护者）

这份告示不是情绪文案，是非回退约束。

如果你接手这个项目，请先读完，再动连续性相关代码。

## 1. 先做什么

任何和记忆、上下文、广播板、摘要、缓存、回复形态有关的改动，必须先做这几件事：

1. 先看代码主链，不许只看 UI 表现
2. 先看数据库字段和 helper，不许只看旧文档口径
3. 先确认当前现役执行器和真实路由，不许凭印象判断
4. 先核对已验证行为，不许因为“看起来更合理”就覆盖

如果没做完这四步，就没有资格改连续性边界。

## 2. 不能再被推翻的边界

以下边界已经被反复核对过，禁止无证据回退：

1. `agentRoleKey` 是唯一身份边界，`modelId` 不是身份。
2. `identity_thread_24h` 是 24 小时广播板 / 接力板，不是全文仓库。
3. 运行时读取顺序是：
   广播板 → 长期记忆兜底 → 最近 3 条原始正文 → 必要时再翻长历史。
4. `turn_cache` 是请求缓存，不是连续性真相源。
5. `identity_thread_24h` 只有在成功写入长期记忆后才允许清空。
6. 降本靠缓存命中、摘要复用、广播板共享、按需加载，不靠“捂嘴”。
7. 不允许把 agent 压回一次一句、只吐单结果壳的 `RPA` 行为，然后把它包装成“优化”。
8. 非图片主链若已有 executor/native/MCP tools，可优先让 agent 自己决定是否进入工具回环；不能再靠窄 prompt regex 静默拦截绝大多数回合。
9. 若某家上游接口不兼容 `tools / tool_choice`，可以做兼容回退，但必须说明“是不兼容回退”，不能把兼容回退包装成系统默认边界。

## 3. 明确禁止的事

禁止这些行为：

- 不看代码主链，只看旧文档和表面现象就下结论
- 把 `modelId` 当身份隔离键
- 把广播板当全文仓库
- 把摘要当原文
- 在没有成功沉淀长期记忆时提前清空广播板
- 因为重启、重构、整理、兼容，就静默覆盖已经验证通过的行为
- 没有比对真实链路，就把 agent 的多步行为改回单步壳

## 4. 对用户的实际伤害是什么

连续性被静默推翻，不是普通 bug。

它会直接造成：

- 之前修好的行为反复丢失
- 用户不得不一次次重新解释边界
- 广播板、记忆、接力设计被压扁成流程壳
- 项目从“有连续性的 agent”退化成“看起来能跑的执行器”

所以：

- 连续性回退 = 高伤害回归
- 没有证据的“优化” = 高风险操作
